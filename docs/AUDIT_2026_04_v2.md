```
┌─────────────────────────────────────────────────┐
│ Layer 1: Application                            │
│   src/main.c          (entry point, 28 LOC)     │

├─────────────────────────────────────────────────┤
│ Layer 2: Manager (orquestador)                  │
│   manager/state.c     (lifecycle, 91 LOC)       │
│   manager/render.c    (UI rendering, 108 LOC)   │
│   manager/input.c     (mouse/keyboard, 200 LOC) │
│   manager/utils.c     (helpers, 17 LOC)         │
├─────────────────────────────────────────────────┤
│ Layer 3: Core Domain (entidades)                │
│   core/workspace.c    (workspace mgmt, 89 LOC)  │
│   core/tab.c          (tab mgmt, 119 LOC)       │
│   core/window.c       (window mgmt, 45 LOC)     │
│   core/logger.c       (ring buffer, 108 LOC)    │

                                              ▼                   ▼

```

**Coupling analysis:**
- ✅ **Cero dependencias circulares** — grafo DAG puro
- ✅ **Unidireccional** — cada capa solo conoce la capa inferior
- ⚠️ **manager conoce todo** — Manager importa workspace, tab, window (diseño intencional de orquestador)
- ⚠️ **types.h es dependency magnet** — todo importa types.h (inevitable con tipos transparentes)

### 2.3 Ownership Model

```
notcurses (external)
    └── TuiManager (calloc en create, free en destroy)
        └── TuiWorkspace[] (calloc en manager, destroy en manager_destroy)
            ├── ws_plane (ncplane, destroy en workspace_destroy)
            └── TuiTab[] (calloc en workspace_create, destroy en workspace_destroy)
                ├── tab_plane (ncplane, destroy en tab_destroy)
                └── TuiWindow[] (calloc en caller, add/remove ownership)
                    ├── plane (ncplane, destroy en window_destroy)
                    └── text_input (widget, destroy en on_destroy callback)
```

**Reglas de ownership:**
1. **Parent owns children**: Manager destruye workspaces, workspace destruye tabs
2. **Windows son transferidas**: `tui_tab_add_window` transfiere ownership al tab
3. **on_destroy fires BEFORE plane destroy**: Previene use-after-free de widgets
4. **Safe realloc**: Siempre usa variable temporal para prevenir leaks en OOM

### 2.4 Data Flow (Input → State → Render Cycle)

```
┌─────────────────────────────────────────────────────────┐
│  Event Loop (60fps, nanosleep 16ms)                     │
│                                                          │
│  1. tui_manager_render(mgr)     ← Full UI repaint       │
│     └─→ notcurses_render(nc)    ← Flush to terminal     │
│                                                          │
│  2. notcurses_get_nblock(nc, &ni) ← Non-blocking input  │
│                                                          │
│  3a. If BUTTON/MOTION:                                   │
│      tui_manager_process_mouse(mgr, key, &ni)           │
│      ├── Tab click → set_active_tab                     │
│      ├── Window click → focus + reorder                 │
│      ├── [X] click → remove_window + destroy            │
│      ├── Grip click → start resize                      │
│      ├── Title drag → start drag                        │
│      └── Motion → resize/drag update                    │
│                                                          │
│  3b. If KEYBOARD:                                        │
│      tui_text_input_handle_key(input, key, &ni)  ← widget│
│      └─ if not consumed →                                │
│         tui_manager_process_keyboard(mgr, key, &ni)     │
│         ├── q/Q → running=false                          │
│         ├── Alt+1..9 → switch workspace                  │
│         ├── ←/→ → cycle tabs                             │
│         ├── Tab → cycle windows                          │
│         ├── x → close window                             │
│         ├── w → close tab                                │
│         └── d → close workspace                          │
│                                                          │
│  4. Loop back to step 1                                  │
└─────────────────────────────────────────────────────────┘
```

**Frame budget**: ~16ms (60fps). Render domina el budget (~5-10ms con notcurses). Input processing es sub-milisegundo.

## 4. Memory Safety Audit

### 4.1 Allocation/Deallocation Map

| Module | Allocations | Deallocations | Balance |
|--------|------------|---------------|---------|
| **Manager** | calloc (manager), calloc (ws array), realloc (ws array) | free (ws array), free (manager), cascading workspace_destroy | ✅ Balanced |
| **Workspace** | calloc (ws), calloc (tabs array), realloc (tabs array) | free (tabs), free (ws), cascading tab_destroy | ✅ Balanced |
| **Tab** | calloc (tab), calloc (windows array), realloc (windows array) | free (windows), free (tab), cascading window_destroy | ✅ Balanced |
| **Window** | calloc (window) | free (window) | ✅ Balanced |
| **Logger** | fopen (file handle) | fclose (file) | ✅ Balanced |
| **Text Input** | calloc (input), ncplane_create | free (input), ncplane_destroy | ✅ Balanced |
| **Demo** | calloc (workspaces, tabs, windows via API) | Delegated to manager destroy | ✅ Balanced |

### 4.2 Realloc Safety

**All 3 realloc sites use the safe pattern:**

```c
// ✅ SAFE: temp variable, check NULL, only assign on success
TuiWindow** new_windows = realloc(tab->windows, new_capacity * sizeof(TuiWindow*));
if (!new_windows) {
    tui_log(LOG_ERROR, "OOM en tui_tab_add_window (realloc)");
    return;  // Original tab->windows still valid
}
tab->windows = new_windows;
tab->window_capacity = new_capacity;
```

**Sites**: `tab.c:64` (add_window), `workspace.c:67` (add_tab), `state.c:48` (add_workspace)

### 4.3 Dangling Pointer Risk Assessment

| Scenario | Risk | Mitigation | Status |
|----------|------|-----------|--------|
| Window destroyed while dragged | 🔴 → ✅ | `mgr->dragged_window = NULL` antes de destroy | ✅ Fixed |
| Window destroyed while resizing | 🔴 → ✅ | `mgr->resizing_window = NULL` antes de destroy | ✅ Fixed |
| Text input destroyed with active ref | 🔴 → ✅ | `on_destroy` callback nullifies `g_active_input` | ✅ Fixed |
| Workspace removed from manager | 🟡 | Manager owns workspace, destroys via cascade | ✅ Safe |
| Tab removed from workspace | 🟡 | Workspace destroys tab, tab destroys windows | ✅ Safe |
| `tui_tab_remove_window` external refs | 🟡 | Function removes from tab but doesn't nullify external refs | ⚠️ **Residual risk** |

### 4.4 Buffer Overflow Analysis

| Buffer | Size | Write method | Safety |
|--------|------|-------------|--------|
| `tab->name[64]` | 64 bytes | `strncpy` + explicit null-term | ✅ Safe |
| `ws->name[64]` | 64 bytes | `strncpy` + explicit null-term | ✅ Safe |
| `TuiLogBuffer.lines[100][256]` | 25,600 bytes | `snprintf` with bounds | ✅ Safe |
| `text_input->buffer[512]` | 512 bytes | `memmove` + bounds check `len < 511` | ✅ Safe |
| Log message formatting | 256 bytes | `vsnprintf` with `sizeof(msg_buf)` | ✅ Safe |

**Assessment**: ✅ **Zero buffer overflow risk**. All writes use bounded functions.

### 4.5 Thread Safety Review

| Resource | Protection | Thread-safe? | Notes |
|----------|-----------|-------------|-------|
| Logger ring buffer | `pthread_mutex_t` | ✅ Yes | Full mutex on all operations |
| Logger log level | Same mutex as buffer | ✅ Yes | `set_level`/`get_level` use mutex |
| `s_next_id` | **None** | ❌ **No** | `int` increment is not atomic |
| notcurses planes | notcurses internal | ⚠️ Depends | notcurses is not thread-safe by design |
| Manager state | Single-threaded app | N/A | TUI apps are inherently single-threaded |

**Issue ID: MEM-TS-01** — `s_next_id` should use `atomic_int` or a mutex for thread safety. Currently, concurrent `tui_*_create` calls from different threads could produce duplicate IDs.

**Severity**: Low for current usage (single-threaded TUI). Medium if used as library in multi-threaded context.

### 4.6 Remaining Memory Issues

| ID | Severity | Issue | Fix |
|----|----------|-------|-----|
| **MEM-01** | 🔴 High | Double logger init: `main.c` calls `tui_logger_init`, then `demo_main` also calls it. First FILE* leaks. | Remove init from `demo_main` |
| **MEM-TS-01** | 🟡 Medium | `s_next_id` not thread-safe | Use `atomic_int` or mutex |
| **MEM-02** | 🟢 Low | `tui_tab_remove_window` doesn't nullify external refs to removed window | Document ownership semantics |

### 5.3 Error Handling Patterns

| Pattern | Usage | Quality |
|---------|-------|---------|
| Return NULL on failure | All factory functions | ✅ Standard C idiom |
| Log error before returning | All failure paths | ✅ Observable |
| Safe no-op on NULL input | All destroy/query functions | ✅ Defensive |
| Early return on invalid params | All functions | ✅ Guard clauses |
| No assertions | Production code | ✅ Runtime checks instead |

### 5.4 Encapsulation Level

| Type | Visibility | Fields exposed | Risk |
|------|-----------|---------------|------|
| `TuiManager` | Transparent (types.h) | All fields | 🟡 Client can mutate internals |
| `TuiWorkspace` | Transparent (types.h) | All fields | 🟡 Client can corrupt state |
| `TuiTab` | Transparent (types.h) | All fields | 🟡 Client can corrupt state |
| `TuiWindow` | Transparent (types.h) | All fields | 🟡 Client can corrupt state |
| `TuiTextInput` | Transparent (text_input.h) | All fields | 🟡 Widget internals exposed |

**Assessment**: ⚠️ **Transparent types are the biggest architectural risk.** Any client can:
- Set `ws->tab_count = -1` (corrupts state)
- Access `mgr->workspaces[0]->tabs[0]->windows[0]->plane` (tight coupling)
- Mutate `tab->active_window_index` directly (bypasses validation)

**Recommendation for v1.0.0**: Convert to opaque types with full getter/setter API. This is a breaking change but essential for library stability.
